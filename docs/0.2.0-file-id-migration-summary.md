# `0.2.0` Stable `file_id` Migration Summary

## Purpose

This document records what was actually implemented during the `0.2.0` migration from filename-based identity to stable `file_id` identity, what was validated during testing, and what remains intentionally out of scope for `0.2.0`.

It is meant to serve two purposes:

- working memory for continued development
- a factual reference for what changed between `0.1.x` and `0.2.0`

## Why The Migration Was Needed

Prior to `0.2.0`, the application effectively treated the filename as the document identity in several workflows:

- document history was keyed by `file_name`
- directory notes were keyed by `file_name`
- checkout/checkin behavior depended heavily on source/locked filenames
- revision access was derived from path-based record keys

That worked until filenames were reused. Once a deleted filename was later reused by a different live document, the new file could inherit the old file's history or notes. In practice, this made rename and delete unreliable in workflows where drawing numbers must be shifted or reused.

## What `0.2.0` Introduced

### Hidden source index

Each tracked source directory now maintains:

- `.doc_control_index.json`

This file is the runtime registry for active and retired document identities within a source directory.

Each indexed document stores:

- `file_id`
- `current_name`
- `canonical_name`
- `status`
- `known_names`
- `legacy_bound_names`
- checkout metadata

### Stable document identity

Tracked source documents now use a stable `file_id` that survives:

- rename
- checkout/checkin cycles
- repeated edits
- note creation
- revision access

### History changes

History entries now support:

- `file_id`
- `previous_file_name`

This allows rename to be represented as a document event rather than a destructive filename rewrite.

### Notes changes

Directory file notes now support:

- `file_id`
- `file_name`

The note's identity follows the document through rename.

### Revision and record migration

Revision lookups and several record-resolution paths were updated to prefer `file_id`, with legacy fallback paths still retained where needed.

### Source-file operations

Source-file rename and delete were reworked around `file_id` rather than filename ownership alone.

## Important `0.2.0` Behavior

### Rename

Rename now:

- preserves the same `file_id`
- appends a `RENAME` event
- records `previous_file_name`
- avoids contaminating the renamed file with the old history of another document that previously used the same filename

### Delete

Delete now:

- retires the document identity rather than erasing it from history
- keeps history and notes attached to the deleted document identity
- prevents a later reused filename from inheriting that deleted document's lineage

### Explorer-added files

Files discovered directly from Explorer can now be indexed without being falsely shown as having document history before a real history event exists.

## Migration / Backfill Logic Added

`0.2.0` includes conservative migration logic for older folders:

- legacy history rows can be backfilled with `file_id`
- legacy notes can be rebound to `file_id`
- history/name matching only backfills when the match is unambiguous
- excluded artifacts such as `.bak`, `.tmp`, and `plot.log` are not treated as managed source files

The migration deliberately avoids aggressive guessing when multiple historical identities could plausibly match the same old filename.

## Testing That Was Performed

Testing was done primarily against copied source folders and scratch folders because this environment did not provide a reliable automated pytest run for all scenarios.

### Core scenarios validated

- add file -> rename -> history continuity
- delete file -> reuse old filename -> no inherited deleted history
- notes follow file identity through rename
- app restart persistence across migrated state
- repeated checkout/checkin on migrated legacy files
- rename -> checkout -> checkin -> restart
- revision snapshots still accessible after rename
- migration from CSV-only legacy folders
- migration from mixed legacy folders that already had JSON-era data

### Real folder results

The best validation cases were:

- copied folders with history only from `0.1.3` / `0.1.4`
- copied folders with legacy CSV-only history
- copied folders where renamed files were edited repeatedly after migration

These cases showed:

- coherent `file_id` lineage
- clean source indexes
- note continuity
- rename continuity
- revision access continuity

### Known special case

Folders already damaged by `0.1.5` filename-identity rename/delete reuse behavior were only partially repairable automatically. Those folders may still contain ambiguous historical rows that cannot be reconstructed perfectly from metadata alone.

## Bugs Found During `0.2.0` Work

The following migration bugs were discovered and fixed during development:

- blank history after add-then-rename due to source-index identity splits
- duplicate active/deleted identities in `.doc_control_index.json`
- Explorer-added files highlighted as if they had history when they did not
- note entries bound to stale or orphaned `file_id`s
- non-source artifacts such as `.bak` and `plot.log` entering the source index

The resulting implementation is materially more stable than the first `0.2.0` migration slice.

## What Was Intentionally Not Completed In `0.2.0`

The following were intentionally left out of `0.2.0`:

- aggressive repair of ambiguous legacy histories
- full automated recovery for metadata already damaged by `0.1.5`
- enabling checked-out rename/delete without restriction
- enabling broader move/copy explorer-style operations under the new identity model
- broader multi-user rollout guarantees

## Operational Recommendation For `0.2.0`

Based on testing completed during development, `0.2.0` is a reasonable branch to pin as the new development baseline for the current small user group.

It should be understood as:

- a stable migration baseline for continued development
- suitable for controlled use by the current active users
- not yet formally proven for broad rollout across all users and all historical edge cases

## Recommended Starting Point For `0.2.1`

`0.2.1` should build on this branch rather than reopening the migration architecture unless a new real defect appears.

Suggested posture:

- treat `0.2.0` migration behavior as the baseline
- address colleague feedback and UX/workflow improvements in `0.2.1`
- only revisit migration logic when a concrete new folder or workflow exposes a real issue
