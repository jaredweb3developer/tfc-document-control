# 0.2.3 Projects Favorites, Local Files, and Notes Logical Grouping

## Feature Request

This update pertains to:

- Projects > Favorites & Local Files
- Projects > Notes

Requested behavior:

- Users should be able to logically group items within each category of `Favorites & Local Files`:
  - `Project Favorites`
  - `Global Favorites`
  - `Checked Out`
  - `Reference Files`
- Users should also be able to logically group entries within `Notes`.
- These groupings should behave like virtual folders:
  - they do not change the real filesystem
  - folders appear at the top of the list
  - folders have clear folder symbology
  - users can enter folders, go up one level, or return to the root view
  - subfolders should be supported

## Review

### Current State

- All affected views are currently flat `QListWidget` lists.
- `Project Favorites` and `Global Favorites` persist as flat path arrays.
- `Project Notes` persist as a flat list of note dictionaries in `dctl.json`.
- `Checked Out` and `Reference Files` are rendered from `CheckoutRecord` rows rather than from a per-view stored list.
- Existing `item_customizations` and group metadata already support styling and ad hoc grouping labels, but they do not support:
  - folder navigation
  - parent/child hierarchy
  - per-folder ordering
  - folder creation/rename/delete

### Key Constraint

This feature should not be built by overloading the existing customization groups feature.

Reason:

- customization groups are descriptive tags/styles
- requested folders are navigable containers with hierarchy and ordering
- combining those concepts would make the data model ambiguous and fragile

### Additional Constraint

`Checked Out` and `Reference Files` need stable item identifiers before folder assignment can be made reliable.

Reason:

- those views are currently keyed by transient list index in several places
- list indexes are not stable enough for persistent folder membership
- record-level logical grouping should be attached to a stable record id instead

## Recommended Design

## UI Strategy

Use a shared logical-folder browser pattern across all requested views instead of five separate one-off implementations.

Recommended UI additions for each affected list:

- a small navigation bar above the list:
  - current folder path or breadcrumb
  - `Up`
  - `Root`
  - `New Folder`
- list rendering that shows:
  - folders first
  - items second
  - clear folder icon/prefix for folder rows
- double-click behavior:
  - folder row enters folder
  - item row keeps its current behavior
- context menu additions:
  - `New Folder`
  - `Rename Folder`
  - `Delete Folder`
  - `Move Selected To Folder`
  - `Move Selected To Root`

This should remain a list-style browser, not a tree widget.

Reason:

- the current UX is already list-based
- folder navigation matches the request directly
- it avoids a broader UI rewrite across the Projects section

## Persistence Strategy

Introduce a dedicated logical-organization model separate from item data.

Recommended model shape per view:

- `folders`
  - folder id
  - folder name
  - parent folder id
  - sort order
- `placements`
  - item key
  - parent folder id
  - sort order

Root is implicit and does not need to be stored as a real folder row.

Recommended storage locations:

- `dctl.json`
  - `project_favorites`
  - `project_notes`
  - `project_checked_out`
  - `project_reference`
- `global_favorites.json`
  - `global_favorites`

Store the logical organization in additive top-level structures such as:

```json
{
  "logical_views": {
    "project_favorites": {
      "folders": [],
      "placements": []
    },
    "project_notes": {
      "folders": [],
      "placements": []
    },
    "project_checked_out": {
      "folders": [],
      "placements": []
    },
    "project_reference": {
      "folders": [],
      "placements": []
    }
  }
}
```

and similarly for `global_favorites.json`.

This keeps:

- favorite paths as favorite paths
- notes as notes
- records as records
- folder organization as a separate concern

## Stable Item Keys

Recommended keys per scope:

- `Project Favorites`: favorite path string
- `Global Favorites`: favorite path string
- `Project Notes`: note id
- `Checked Out`: new persistent record id
- `Reference Files`: new persistent record id

Before grouping `Checked Out` and `Reference Files`, add a persistent `id` field to `CheckoutRecord` and preserve it through save/load.

## Rendering Rules

For each logical-folder-enabled view:

1. Determine current folder id.
2. Render child folders of that folder first.
3. Render items placed directly in that folder second.
4. Exclude items assigned to other folders.
5. If a search term is active, search should include:
   - item display text
   - item metadata already searched today
   - folder names

Recommended search behavior for MVP:

- when search is empty, show only current folder contents
- when search is active, flatten matching descendants into the current result set and include enough context in the tooltip to show the folder path

That avoids the common failure mode where users “lose” items because they forgot which folder they are in.

## Build Plan

### Phase 1: Data Model and Persistence

Goal:

- add the persistence primitives without changing visible behavior yet

Tasks:

- add a reusable logical-folder schema and normalizers
- extend project config load/save with `logical_views`
- extend global favorites load/save with `logical_views`
- add `CheckoutRecord.id` and make save/load backward compatible
- create helpers for:
  - folder CRUD
  - placement CRUD
  - ordering within a folder
  - folder path resolution

Success criteria:

- old configs still load
- new configs save without breaking older fields
- every record has a stable id after load, even when loaded from legacy data

### Phase 2: Shared Browser Controller

Goal:

- avoid duplicating folder-navigation logic in five separate views

Tasks:

- build a shared helper/controller for logical folder lists
- define a small adapter interface per scope:
  - list widget
  - search widget
  - item source function
  - item display label function
  - item key function
  - open/edit/remove handlers
- add support for:
  - current folder state
  - folder-first rendering
  - folder icons/symbology
  - up/root navigation

Success criteria:

- one implementation path can render at least two scopes without copy/paste logic

### Phase 3: Favorites Integration

Goal:

- enable logical folders for:
  - `Project Favorites`
  - `Global Favorites`

Tasks:

- wire folder browsing into favorites lists
- add folder actions to options/context menus
- keep existing favorite add/open/remove actions working
- ensure reordering works within the current folder, not just root

Success criteria:

- favorites can be placed into folders/subfolders
- users can still open favorites normally
- folder operations do not alter actual file locations

### Phase 4: Notes Integration

Goal:

- enable logical folders for `Project Notes`

Tasks:

- wire project notes into the shared folder browser
- ensure note create/edit/remove still works from nested folders
- decide whether copied/moved notes land:
  - in root by default
  - or in the currently open folder

Recommended default:

- new or transferred notes land in the currently open folder when that view is active
- otherwise they land in root

Success criteria:

- nested note organization works
- note editing remains unchanged except for navigation context

### Phase 5: Checked Out and Reference Files Integration

Goal:

- enable logical folders for record-backed views

Tasks:

- switch checked-out and reference folder membership to stable record ids
- update list rendering to use logical-folder placements
- verify record actions still target the correct underlying record after grouping
- define behavior for deleted/checked-in records:
  - remove stale placements automatically when the backing record disappears

Success criteria:

- grouping survives refreshes and record list reordering
- removing or checking in a record does not leave broken UI state

### Phase 6: Polish and Migration Cleanup

Goal:

- make the experience understandable and resilient

Tasks:

- add folder tooltips and empty-state messaging
- confirm search behavior across nested folders
- decide whether empty folders can be deleted only when empty or with cascade

Recommended default:

- allow recursive delete with confirmation
- move contained items to root only if explicitly requested in a future enhancement

- add safeguards for duplicate folder names under the same parent
- add migration cleanup for orphaned placements and invalid parent ids

## Risks

### Medium Risk

- record grouping will be brittle if record ids are not introduced first
- search can become confusing if it is folder-scoped without a clear fallback
- ordering behavior can feel inconsistent if root and nested folders use different rules

### Low Risk

- additive persistence fields should be safe if loaders normalize missing data cleanly
- the current list-based UI can support this without a full widget rewrite

## Recommended Delivery Order

1. Add record ids and logical-view persistence.
2. Build the shared folder browser helper.
3. Ship `Project Favorites` and `Project Notes` first.
4. Extend to `Global Favorites`.
5. Extend to `Checked Out` and `Reference Files`.
6. Polish search, migration cleanup, and edge cases.

## Out of Scope for Initial Build

- changing real filesystem structure
- syncing logical folders with disk folders
- drag-and-drop folder management
- cross-scope shared folders
- applying the same feature to `Global Notes` unless requested separately

## Recommendation

This is a good `0.2.3` feature, but it should be treated as a small organization framework rather than a set of isolated UI tweaks.

If built around:

- stable item keys
- dedicated logical-view persistence
- one shared folder-browser implementation

then it should scale cleanly across all requested scopes without becoming hard to maintain.
