# 0.2.3 Reference Files Refresh And Update Plan

## Goal

Add a reference-refresh workflow in `Projects > Favorites & Local Files > Reference Files` so users can:

- refresh selected reference copies from their source files
- refresh selected reference copies only when safe
- run an `Update All References` workflow for the current project
- make per-file replacement decisions
- optionally apply one decision to all remaining applicable files

## Working Build Notes

This document is the active implementation reference for the feature build.

Use it as the working checklist while the implementation is in progress.

Current intent:

- plan first
- implement in additive slices
- keep all on-disk record formats backward compatible
- preserve current `Reference Files` logical-folder behavior while refresh features are added

Not in scope for the first build unless the implementation forces it:

- three-way merge behavior
- diff or preview UI
- automatic recreation of missing local references without user confirmation
- cross-project bulk reference management

## Desired User-Facing Features

### Reference Actions

Add actions for:

- `Refresh Selected Ref`
- `Refresh Selected Ref (If Unchanged)`
- `Check Reference Status`
- `Update All References`

Recommended placements:

- `Reference Files` tab options menu
- `Reference Files` context menu
- optionally `Checked Out Files > Reference Copies` later if that view needs parity

### Update All Workflow

`Update All References` should:

1. inspect all reference-copy records for the current project
2. compare each reference against its source and local copy state
3. present statuses in a review dialog
4. allow the user to choose per file:
   - `Replace Local With Source`
   - `Keep Local Reference`
   - `Skip`
5. support:
   - `Apply this choice to remaining files`
   - preferably `Apply this choice to remaining files with the same status`

## Current State Review

### Existing Reference Copy Data

Today each `reference_copy` record already stores:

- `source_file`
- `local_file`
- `project_dir`
- `source_root`
- `record_type`
- `file_id` for source-backed references when available

This is enough for a basic overwrite refresh.

### Current Limitation

The app does not currently persist enough metadata to safely determine:

- whether the source changed since the copy was made
- whether the local reference copy changed since the copy was made
- whether a refresh is safe or potentially destructive

That means a robust update workflow needs additional metadata.

## Concrete Build Strategy

Build this in four code-facing layers:

1. Persist enough baseline metadata on each `reference_copy` record to reason about change safely.
2. Add a deterministic status engine that classifies one reference record at a time.
3. Expose simple selected-item refresh actions on the existing `Reference Files` view.
4. Add a project-level `Update All References` review dialog that uses the same status engine and execution helpers.

This keeps the status logic reusable and avoids burying file-state decisions inside the dialog code.

## Recommended Data Model Changes

Extend `CheckoutRecord` for `reference_copy` rows with additive fields:

- `source_hash_at_copy`
- `local_hash_at_copy`
- `source_mtime_at_copy`
- `local_mtime_at_copy`
- `last_refreshed_at`

Optional but recommended:

- `source_size_at_copy`
- `local_size_at_copy`

Recommended rule:

- store these fields for all new reference copies
- backfill as empty values for existing records
- keep load/save backward compatible

Recommended minimum shape for a `reference_copy` row after this change:

```json
{
  "id": "rec_...",
  "record_type": "reference_copy",
  "source_file": "V:/Clients/.../file.pdf",
  "local_file": "C:/Users/.../Projects/.../Reference/file.pdf",
  "source_hash_at_copy": "sha1...",
  "local_hash_at_copy": "sha1...",
  "source_mtime_at_copy": 1775637821.123,
  "local_mtime_at_copy": 1775637821.456,
  "source_size_at_copy": 58214,
  "local_size_at_copy": 58214,
  "last_refreshed_at": "2026-04-08T12:00:00"
}
```

Notes:

- exact timestamp formatting can follow the existing record serialization style
- additive empty-string or zero-value defaults are acceptable for legacy rows

## Status Model

Each reference should be classified into one of these statuses:

- `up_to_date`
  - source appears unchanged and local reference appears unchanged since copy or refresh
- `source_changed_safe`
  - source changed, local reference unchanged
  - safe default: replace
- `local_changed_only`
  - local reference changed, source unchanged
  - safe default: keep local
- `both_changed_conflict`
  - source changed and local reference changed
  - safe default: skip unless user explicitly replaces
- `source_missing`
  - source file no longer exists
- `local_missing`
  - local reference file no longer exists
- `untracked_state`
  - older reference record lacks enough metadata for safe classification

Recommended planned-action enum:

- `none`
- `replace`
- `keep`
- `skip`

Recommended default action map:

- `up_to_date` -> `none`
- `source_changed_safe` -> `replace`
- `local_changed_only` -> `keep`
- `both_changed_conflict` -> `skip`
- `source_missing` -> `skip`
- `local_missing` -> `skip`
- `untracked_state` -> `skip`

## Refresh Semantics

### `Refresh Selected Ref`

Behavior:

- overwrite selected local reference copy with current source file
- skip if source is missing
- refresh metadata after successful copy

Use when:

- user wants a direct latest-source pull and accepts overwrite

### `Refresh Selected Ref (If Unchanged)`

Behavior:

- only refresh selected references when local reference is unchanged from the last copied or refreshed baseline
- skip files with local modifications or missing baseline metadata

Use when:

- user wants a safe pull-latest behavior without overwriting local edits

### `Check Reference Status`

Behavior:

- inspect selected references or all current-project references
- show the current classification without modifying files

Use when:

- user wants to understand what will happen before refreshing

### `Update All References`

Behavior:

- inspect all current-project reference copies
- open a review dialog with statuses and planned actions
- allow the user to:
  - approve safe replacements in bulk
  - skip conflicts
  - override individual actions
  - apply one decision to remaining rows

Recommended row actions:

- read-only details text for why the row received that status

Recommended status defaults:

- `up_to_date` -> `None`
- `source_changed_safe` -> `Replace`
- `local_changed_only` -> `Keep`
- `both_changed_conflict` -> `Skip`
- `source_missing` -> `Skip`
- `local_missing` -> `Skip`
- `untracked_state` -> `Skip`

### Review Dialog Instead Of Repeated Popups

Do not implement this as one modal prompt per file.

Reason:

- too slow for many references
- hard to review consistently
- awkward to apply one choice to many rows

A review table is the right implementation.

## Technical Notes

### Hashing Recommendation

Use content hashes for reliable comparison.

Recommended:

- SHA-1 is acceptable here for change detection and matches existing lightweight usage
- no security requirement is implied

### Why Hashes Over Timestamps Alone

Timestamps are not enough because:

- copied files may preserve timestamps
- timestamps can drift during network or external file operations
- size plus mtime is better than mtime alone, but still weaker than a hash

Recommended approach:

- store both timestamp and hash
- use hash as primary comparison when available
- use timestamp and size as fallback if hashing fails

### Existing Record Removal And Check-In Interactions

Reference refresh should not interfere with:

- reference removal
- checked-out file workflows
- logical-folder placement for reference rows

Important:

- record `id` must remain stable when refreshing a reference
- `local_file` should not change during a normal refresh
