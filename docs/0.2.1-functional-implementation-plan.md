# `0.2.1` Functional Implementation Plan

## Purpose

This document defines the agreed functional scope for `0.2.1` and provides the implementation and testing plan for those changes.

This version deliberately excludes layout-only or visual redesign discussion and focuses on functional behavior.

## Agreed `0.2.1` Scope

The current `0.2.1` functional scope is:

1. Tracked directory move / relink support
2. Source-file details in the Files area
3. Project note copy / move between projects

## Why These Items Were Chosen

### 1. Tracked directory move / relink support

This is the most important functional gap because a moved or renamed tracked directory can break project configuration even when the internal document-control data is still valid.

### 2. Source-file details in the Files area

This is low-risk and useful. File size and date metadata come from the filesystem and do not conflict with the new `file_id` model.

### 3. Project note copy / move between projects

This is practical and lower risk than transferring file notes because project notes are not tied to `file_id`.

## Explicitly Out Of Scope For `0.2.1`

The following are not part of this document unless later added explicitly:

- general layout or resizing changes
- pure UI-only redesign requests
- file-note transfer between files or projects
- individual file relink after external file move
- checked-out file move/copy workflows
- broad migration redesign beyond `0.2.0`

## Functional Item 1: Tracked Directory Move / Relink Support

## Problem

If a tracked source directory is moved or renamed outside the application, the stored project configuration can still point at the old path. The app then treats the source as missing even though the directory still exists elsewhere.

## `0.2.1` Goal

Allow the user to relink a missing tracked source directory to a new location without losing document-control continuity.

## Recommended `0.2.1` Behavior

### Minimum supported behavior

When a tracked source directory no longer exists:

- show it as missing
- allow the user to choose a replacement directory
- verify that the replacement looks like the same tracked source
- update project configuration to the new path

### Validation rules

The replacement directory should be accepted if one of these is true:

- it contains a matching `.doc_control_index.json`
- it contains matching document-control history for the same source
- it is explicitly accepted by the user after a mismatch warning

For `0.2.1`, the safest initial rule is:

- prefer exact metadata match when available
- otherwise require a confirmation warning

### Data that should remain unchanged

Relinking should not rewrite:

- source history
- source notes
- source index identities
- checkout history entries

Relinking should only update:

- project configuration source path
- any stored source-root mapping keyed by source path, if applicable

## `0.2.1` Implementation Approach

### Initial version

Implement manual relink only.

Do not attempt automatic folder searching in `0.2.1`.

Reason:

- lower risk
- easier to validate
- easier to explain to users

### Suggested workflow

1. App detects missing tracked source directory
2. User chooses `Relink Source Directory...`
3. User selects new folder
4. App validates metadata compatibility
5. App updates stored source path
6. App refreshes source browser and file list

## Test cases

### Basic relink

- Track a source folder
- Close app or leave open
- Rename or move the source folder outside the app
- Use relink
- Confirm history, notes, and source list still work

### Restart persistence

- Relink a moved folder
- Close app
- Reopen app
- Confirm the new path persists

### Metadata mismatch warning

- Attempt relink to an unrelated folder
- Confirm the app warns appropriately

## Functional Item 2: Source-File Details In Files Area

## Problem

The Files area currently shows filename only. Users want quick filesystem context such as:

- modified date
- file size

## `0.2.1` Goal

Expose useful file metadata in the source-file listing without changing document identity behavior.

## Recommended `0.2.1` Behavior

### Minimum supported fields

- file size
- last modified timestamp

Creation date can be deferred because it is less consistently useful and can be noisier across filesystems.

### Behavior rules

- metadata is read directly from the current filesystem item
- metadata should not be persisted in document-control files
- metadata should refresh with normal source-file refresh behavior

## `0.2.1` Implementation Approach

The best implementation likely requires changing the Files area from a simple single-column list to a structured multi-column view or equivalent detail presentation. The exact widget choice is an implementation detail, but the functional behavior should be:

- filename remains the primary identifier
- details are shown alongside it
- existing history styling and file selection behavior remain intact

## Test cases

### Metadata display

- Open a source folder with files of different sizes and dates
- Confirm size and modified date display correctly

### Refresh behavior

- Modify a file outside the app
- Refresh the folder
- Confirm size/date update

### History styling compatibility

- Confirm checked-out and historical files still show the correct state styling

## Functional Item 3: Project Note Copy / Move Between Projects

## Problem

Users want to reuse or transfer project notes between projects.

This feedback applies to project notes, not file notes.

## `0.2.1` Goal

Allow users to copy or move project notes from one project to another.

## Recommended `0.2.1` Behavior

### Supported operations

- `Copy Note To Project...`
- `Move Note To Project...`

### What should transfer

- subject
- body
- created-by metadata
- created timestamp
- updated timestamp

### Thread behavior

If the selected note has replies, the whole thread should move or copy together.

Reason:

- moving or copying only the parent note while leaving replies behind would corrupt the thread structure

### Move semantics

For move:

- write the copied thread to the target project first
- only remove the source thread after successful write

### Copy semantics

For copy:

- preserve original content and authorship metadata
- generate new note IDs in the target project
- preserve parent/child structure using remapped IDs

## `0.2.1` Implementation Approach

### First version

Implement project-to-project note transfer only.

Do not attempt:

- project note to file note transfer
- file note to project note transfer
- cross-scope note merging

## Test cases

### Simple copy

- Create a project note
- Copy it to another project
- Confirm both projects now contain the note

### Simple move

- Move a project note to another project
- Confirm it exists only in the destination afterward

### Threaded note copy

- Create a note with replies
- Copy it to another project
- Confirm the thread structure is preserved

### Threaded note move

- Move a note thread to another project
- Confirm the full thread moves intact

### Restart persistence

- Perform copy/move
- Restart app
- Confirm note data remains correct

## Recommended Build Order

Implement in this order:

1. Tracked directory move / relink support
2. Project note copy / move between projects
3. Source-file details in the Files area

## Deferred For `0.2.2`

The following item was discussed during `0.2.1` planning but is intentionally deferred:

### Individual file relink after external file move

Problem:

- if a managed source file is moved externally rather than through the app, the current system does not provide a safe file-level relink workflow

Why deferred:

- this is separate from tracked directory relink
- it likely requires path-aware source indexing beyond the current per-directory filename model
- it should probably be introduced as a first-class document-control move workflow rather than an ad hoc recovery tool

Recommended future direction:

- treat file move as a managed document-control operation
- extend source identity tracking to support relative-path changes within a tracked source root
- add a `MOVE` history event once path continuity can be preserved safely

## Why This Order

- Relink support is the most operationally important
- Project note transfer is functionally meaningful and mostly isolated
- File details are useful but likely touch more UI structure, so they can come after the higher-risk persistence work

## Validation Strategy

Each feature should be validated first on copied project/source folders, not live working folders.

### Recommended validation sequence

1. Implement one feature
2. Test on copied data
3. Restart app and retest persistence
4. Inspect affected JSON/config files directly
5. Only then move to the next feature

## Success Criteria For `0.2.1`

`0.2.1` is successful if:

- missing tracked source folders can be manually relinked without losing continuity
- project notes can be copied or moved between projects safely
- file size and modified date can be viewed in the Files area
- none of the above regress `0.2.0` document-control identity behavior
